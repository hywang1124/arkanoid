# Arkanoid C++ 期末项目大纲

## 项目主题

项目名称：Arkanoid / Breakout 打砖块游戏

本项目使用 C++ 实现经典打砖块游戏。玩家控制屏幕底部的挡板，让小球反弹并击碎砖块，通过得分、生命值和关卡目标完成游戏。

## 协作与语言规则

- `AGENTS.md` 和项目说明文档使用中文编写。
- C++ 源代码中的注释必须使用日语编写。
- 类名、函数名、变量名建议使用清晰的英文命名。
- README 可以使用中文为主，必要时补充英文术语。
- PPT 使用日文，但必须清楚说明项目内容、成员姓名和学生 ID。

代码注释示例：

```cpp
// ボールの位置を更新する
void updateBall();

// パドルが画面外に出ないように制限する
void clampPaddlePosition();
```

## 项目目标

- 完成一个可以实际运行和演示的 C++ 打砖块游戏。
- 通过类和对象展示面向对象编程思想。
- 使用清晰的文件结构和可读性较好的代码风格。
- 实现碰撞检测、计分、生命值、游戏状态和关卡流程。
- 准备 PowerPoint 发表资料和 10 分钟内可以完成的实演 Demo。

## 主要功能

### 必做功能

- 使用键盘控制挡板左右移动
- 小球自动移动
- 小球与墙壁、挡板、砖块发生碰撞后反弹
- 多个砖块按行列排列
- 击碎砖块后增加分数
- 玩家拥有生命值
- 判断游戏胜利和游戏失败
- 支持重新开始或再次游玩
- 使用控制台或图形窗口显示游戏画面

### 可选扩展功能

- 多关卡
- 不同耐久度的砖块
- 道具系统，例如加宽挡板、增加生命、降低球速
- 音效
- 开始菜单和暂停菜单
- 最高分保存与读取
- 简单动画或视觉效果

## 技术选择

- 使用 C++ 配合 SFML、SDL2 或 raylib
- 使用窗口绘制游戏对象
- 支持实时键盘输入
- 画面更适合发表和演示
- 技术表现力更强

## 核心类设计

### Game 类

职责：

- 管理主游戏循环
- 管理游戏状态
- 处理玩家输入
- 更新游戏对象
- 处理碰撞
- 绘制游戏画面

可能包含的成员：

- currentState
- score
- lives
- level
- paddle
- ball
- bricks

### Paddle 类

职责：

- 保存挡板的位置和大小
- 控制挡板左右移动
- 限制挡板不离开游戏边界
- 提供碰撞检测所需的范围信息

### Ball 类

职责：

- 保存小球的位置、方向和速度
- 每一帧更新小球位置
- 与墙壁、挡板、砖块碰撞后改变方向
- 玩家失误后重置小球位置

### Brick 类

职责：

- 保存砖块的位置、大小和耐久度
- 判断砖块是否仍然存在
- 被小球击中后减少耐久度或消失
- 被破坏时增加分数

### Level 类

职责：

- 生成砖块布局
- 重置关卡
- 判断是否清除了所有砖块
- 在后续关卡中提高难度

### ScoreManager 类

职责：

- 记录当前分数
- 可选：保存和读取最高分
- 在游戏中显示分数

## 游戏状态

建议设计以下状态：

- 开始菜单
- 游戏中
- 暂停
- 当前关卡完成
- 游戏失败
- 全部通关

这样可以让游戏流程更清楚，也方便在发表时解释程序结构。

## 实现计划

### 第一步：项目基础设置

- 创建 C++ 项目结构。
- 确认编译器和构建方式。
- 决定使用控制台、SFML、SDL2 还是 raylib。
- 创建最小主循环。

### 第二步：实现挡板和小球

- 绘制挡板和小球。
- 使用键盘移动挡板。
- 让小球自动移动。
- 实现小球与墙壁的碰撞。

### 第三步：实现砖块系统

- 创建砖块网格。
- 检测小球与砖块的碰撞。
- 砖块被击中后消失或减少耐久。
- 击碎砖块后增加分数。

### 第四步：实现游戏规则

- 添加生命值。
- 玩家漏接小球后重置小球。
- 所有砖块被清除后判定胜利。
- 生命值为 0 时判定游戏失败。

### 第五步：完善体验

- 添加标题画面、暂停、重新开始。
- 优化碰撞手感。
- 调整球速和难度。
- 根据时间添加音效、关卡、道具或最高分功能。

### 第六步：准备发表和 Demo

- 制作 PowerPoint。
- 准备短时间内稳定展示的 Demo 流程。
- 发表前反复测试项目。
- 将说明控制在 10 分钟内。

- 头文件和源文件分离

## 技术难点

发表中可以讨论以下问题：

- 小球撞到砖块角落时，反弹方向可能不自然。
- 挡板反弹需要有一定手感。
- 球速太慢会无聊，太快会难以演示。
- 游戏状态增多后，流程管理会变复杂。
- 控制台实时输入和刷新可能比较麻烦。
- 如果使用 SFML、SDL2 或 raylib，外部库配置可能花费时间。

## 解决思路

可以在发表中说明以下解决方法：

- 将碰撞检测拆成独立函数。
- 使用面向对象设计，让每个类只负责自己的功能。
- 先测试小球移动，再逐步加入墙壁、挡板和砖块碰撞。
- 使用固定帧率或时间间隔，让移动速度稳定。
- 先完成最小可玩版本，再添加扩展功能。
- Demo 版本以稳定为优先，不在最后阶段加入高风险功能。
